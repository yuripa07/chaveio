# i18n Guide

Locales supported: **pt-BR** (default) and **en**.

---

## Architecture

- No external library. Pure React Context + typed translation objects.
- `src/locales/translations.ts` — single source of truth for all UI strings.
- `src/contexts/locale-context.tsx` — `LocaleProvider` + `useLocale()` hook.
- `src/components/locale-switcher.tsx` — fixed bottom-right PT/EN toggle button.
- `src/lib/translate-api-error.ts` — maps English API errors to the active locale.

---

## Adding a new locale string

**1. Add the key to the `Translations` interface** in `src/locales/translations.ts`:

```ts
export interface Translations {
  myPage: {
    title: string;
    itemCount: (n: number) => string; // use a function for interpolated strings
  };
}
```

**2. Add the pt-BR value** to the `ptBR` object:

```ts
export const ptBR: Translations = {
  myPage: {
    title: "Minha página",
    itemCount: (n) => `${n} itens`,
  },
};
```

**3. Add the English value** to the `en` object:

```ts
export const en: Translations = {
  myPage: {
    title: "My page",
    itemCount: (n) => `${n} items`,
  },
};
```

**4. Use in the component:**

```tsx
"use client";
import { useLocale } from "@/contexts/locale-context";

export default function MyPage() {
  const { t } = useLocale();
  return <h1>{t.myPage.title}</h1>;
}
```

## Namespaces

Each page/domain has its own namespace in the `Translations` interface. Current namespaces:

| Namespace | Used by |
|---|---|
| `common` | Shared strings (back, loading, kick, you, etc.) |
| `landing` | `app/page.tsx` |
| `createTournament` | `app/tournament/new/page.tsx` |
| `lobby` | `app/tournament/[code]/page.tsx` |
| `lobbyCTA` | `components/lobby-cta.tsx` |
| `bracket` | `app/tournament/[code]/bracket/page.tsx` |
| `live` | `app/tournament/[code]/live/page.tsx` |
| `results` | `app/tournament/[code]/results/page.tsx` |
| `rankingsTable` | `components/rankings-table.tsx` |
| `bracketView` | `components/bracket-view.tsx` |
| `auth` | `components/google-sign-in-button.tsx`, `components/user-chip.tsx`, auth CTAs in `app/page.tsx`, `app/tournament/new/page.tsx`, and `app/tournament/[code]/page.tsx` |
| `apiErrors` | Looked up by `translateApiError()` — see below |

When adding a new page, add a new namespace. When adding a string used across multiple pages, add it to `common`.

## Translating API errors

**Rule: API routes always return error strings in English.** The frontend translates them.

```ts
// ✅ Correct — API route
return NextResponse.json({ error: "Tournament not found" }, { status: 404 });

// ❌ Wrong — never return pt-BR from the API
return NextResponse.json({ error: "Torneio não encontrado" }, { status: 404 });
```

**On the frontend**, wrap any `body.error` in `translateApiError`:

```tsx
import { translateApiError } from "@/lib/translate-api-error";

const body = await response.json();
setError(translateApiError(body.error, t) ?? t.myPage.genericError);
```

`translateApiError` looks up `body.error` in `t.apiErrors`. For pt-BR it returns the Portuguese translation; for English it returns the English string unchanged (since `en.apiErrors` is `{}`). If the key is unknown, it falls back to the raw English string.

**When adding a new API error message**, add the English string to the API route **and** add a pt-BR entry to `ptBR.apiErrors` in `translations.ts`:

```ts
// In the API route:
return NextResponse.json({ error: "Item limit exceeded" }, { status: 422 });

// In ptBR.apiErrors:
"Item limit exceeded": "Limite de itens excedido",
```

**Dynamic errors** (template literals like `` `Tournament must have exactly ${n} round themes` ``) cannot be static keys in `apiErrors`. Leave them in English — they are edge cases that won't appear in normal flows.

## Locale detection and storage

- On first visit, locale is detected from `navigator.language` (`"pt-BR"` → `ptBR`, anything else → `en`).
- Persisted in `localStorage` key `"chaveio:locale"`.
- `document.documentElement.lang` is updated on every locale change.
- SSR renders without locale (hydration-safe): components only get translations after mount.

---

## Rules

- **Never hardcode UI strings** in components. Always use `t.*`.
- **Never hardcode pt-BR strings in API routes.** Always English.
- **Always add both pt-BR and en** when adding a new key — the `Translations` interface enforces this at compile time.
- Components that need translations must be `"use client"` and call `useLocale()`.
- Server components cannot use `useLocale()`. If a server component needs a string, pass it as a prop from a client parent.

---

## Integration tests

Integration tests call API routes directly and assert on English error strings:

```ts
// ✅ Correct
expect(body.error).toMatch(/picks/i);

// ❌ Wrong — API returns English now
expect(body.error).toMatch(/palpites/i);
```
